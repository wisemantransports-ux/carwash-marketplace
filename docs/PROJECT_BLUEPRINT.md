# AutoLink Africa: Project Blueprint & Study Guide

This document provides a technical overview of the platform's architecture, data patterns, and logic flows. It is designed for developers to understand how the system scales and maintains data integrity.

---

## 1. Core Architecture
- **Framework**: Next.js 15 (App Router)
- **Database/Auth**: Supabase (PostgreSQL + GoTrue)
- **AI Engine**: Genkit (Gemini 2.5 Flash)
- **Styling**: Tailwind CSS + Shadcn UI

---

## 2. Authentication Strategy (Dual-Flow)
The system uses two distinct authentication paths to balance security for operators and frictionless access for consumers.

### A. Partners & Admins (Email/Password)
- **Location**: `src/app/login/page.tsx` (Partner Tab)
- **Logic**: Traditional credentials stored in Supabase Auth.
- **Redirection**: Middleware/Layout checks the `role` column in the `users` table to route to `/admin/*` or `/business/*`.

### B. Customers (WhatsApp OTP)
- **Location**: `src/app/login/page.tsx` (Customer Tab)
- **Logic**: 
  1. User enters WhatsApp number.
  2. System generates a 6-digit code and updates the `users` record.
  3. Upon verification, the user is signed in **Anonymously** with metadata identifying them as a customer.
- **Goal**: Frictionless lead capture.

## 3. The "Manual Wiring" Pattern
To ensure 100% reliability in a Supabase/RLS environment, the marketplace avoids deep SQL joins which can often return `null` unexpectedly.

### The 3-Stage Fetch:
1. **Stage 1 (The Filter)**: Fetch IDs of businesses that meet criteria (e.g., `verification_status = 'verified'`).
2. **Stage 2 (The Fetch)**: Use those IDs to fetch listings from the `listings` table using the `.in()` filter.
3. **Stage 3 (The Wiring)**: Map business metadata (Name, City, Logo) back to the listing objects in JavaScript memory before rendering.

## 4. Data Schema Highlights
- **`users`**: Central profile store. Includes `paid` and `trial_expiry` for access control.
- **`businesses`**: Professional profiles. Includes `category` (Wash, Spare, Cars) which determines UI placement.
- **`listings`**: Unified table for all products.
  - `listing_type`: 'car', 'spare_part', or 'wash_service'.
  - `image_url`: Singular string for the primary listing photo (Legacy `images` array is deprecated).
- **`leads`**: Connects customers to sellers. Designed for "Progressive Identity"—creating a user record automatically upon the first inquiry.

---

## 5. Subscription & Access Guards
Access to professional tools is gated at the **Layout Level** (`src/app/business/layout.tsx`).

- **Logic**: Checks `(paid === true) OR (trial_expiry > now)`.
- **Behavior**: If a business is inactive, the dashboard UI is replaced by a "Restricted Access" screen, blocking all operations except Profile and Subscription management.
- **Trial**: New accounts receive a 14-day trial automatically upon registration.

## 6. Operation Flows

### Lead Capture Flow
1. **Discovery**: Customer finds a car or part in the marketplace.
2. **Inquiry**: Customer enters Name/WhatsApp in the `LeadModal`.
3. **Identity**: System checks if the WhatsApp number exists; if not, it creates a "Customer" user.
4. **Persistence**: A record is saved to the `leads` table.
5. **Coordination**: The UI opens a direct WhatsApp chat with the seller pre-filled with listing details.

### Car Wash Operational Flow
1. **Request**: Customer submits a booking via `BookingModal`.
2. **Visibility**: Business sees the request in the "Operations" tab (`wash_bookings` table).
3. **Assignment**: Business **must** assign an `employee` before the booking can be confirmed.
4. **Completion**: Once finished, the customer can "Finish & Rate" the service, which updates the business's public rating.

## 7. Scaling Bottlenecks (Known)
- **Manual Verification**: Admins must manually review Mobile Money screenshots. This is efficient for <100 partners but requires automated gateway integration (e.g., DPO or FNB Pay) for large-scale growth.
- **Interleaving**: In-memory sorting of 1000+ listings may impact client-side performance; future versions should use a database View for interleaved discovery.
